# 기존 GPTs 인스트럭션과 현재 Web App 의 상세 비교·대조 (Part 1)

현재 개발된 Scenario Self Generation Agent 웹 앱은 기존 ChatGPT 플랫폼 기반의 GPTs 환경(Prompt Engineering)에서 겪었던 한계점들을 극복하고, 마케터 환경에 맞춘 완전 독립형 솔루션으로 진화했습니다.

아래는 각 항목별(구성, 사용자 방식 등) 기존 프롬프트 방식과 현 웹앱의 비교대조 및 웹앱의 특장점입니다.

---

## 1. 구성 요소 및 아키텍처 비교 (Architecture / Structure)

**[기존 GPTs 방식]**
- **기반 기술:** ChatGPT 플랫폼 내 챗봇 (자연어 프롬프트). 
- **설계 구조:** `[Part 1~7]` 로 나뉜 방대한 텍스트 인스트럭션을 통해 AI에게 제약사항과 로직을 강요하는 방식입니다. DB 데이터(Explore Contents)를 파일로 업로드하여 RAG(Retrieval-Augmented Generation) 방식으로 검색을 지시했습니다.
- **제약:** 지시사항이 너무 길면 AI가 일부 규칙을 잊어버리거나(Context Window 한계), RAG 시스템 특성상 정해진 DB를 무시하고 임의로 창작(Hallucination)하는 문제가 빈번히 발생했습니다.

**[현재 Web App 방식]**
- **기반 기술:** Node.js (Backend) + HTML/CSS/Vanilla JS (Frontend) 기반의 독립 애플리케이션.
- **설계 구조:** 텍스트가 아닌 '코드 논리(Algorithm)'로 룰을 통제합니다. 
- **특장점 (웹앱의 압승):** 프롬프트 하나에 모든 동작을 맡기지 않고, '매칭 영역'과 '생성 영역'을 분리했습니다. AI는 창작 텍스트 생성에만 사용되고 핵심 매칭 모델은 100% 자바스크립트 수식(`curation-engine.js`)으로 고정 구동되어 **환각 오답 확률이 0%**가 되었습니다.

---

## 2. 사용자(마케터) 방식과 단계별 경험 (User Interaction / UX)

**[기존 GPTs 방식]**
- **방식:** 대화형 인터페이스 (Chat UI).
- **흐름:** 사용자가 채팅창에 "Q1은 1번, Q2는 한국, Q3는 1인가구..." 같이 타이핑을 해야 진행되었습니다.
- **단점:** AI가 중간에 멋대로 4단계를 건너뛰어 바로 결과물을 내놓거나, 오타가 나도 확인 없이 임의로 해석하는 등 **진행 통제가 불가능**했습니다. 접근 권한(코드) 확인 역시 채팅으로 회유가 가능해 취약점이 존재했습니다.

**[현재 Web App 방식]**
- **방식:** GUI 기반의 4-Step 마법사(Wizard) 인터페이스.
- **흐름:** 사용자는 키보드를 칠 필요 없이 제공되는 시각적 '카드'와 '버튼'을 클릭하여 입력합니다. (국가 선택 Dropdown, 기기 체크리스트 등)
- **특장점:** 
    1. **시각적 편리함과 오류 방지:** 클릭과 터치로 진행하므로 무효한 값이 서버로 넘어갈 확률이 근본적으로 차단됩니다.
    2. **동적 카탈로그 (`LOCALE_CATALOGS`):** 국가 선택에 따라 판매되는 삼성 기기와 파트너 기기 라인업이 실시간으로 교체됩니다. 이는 챗봇 프롬프트로는 구현하기 불가능하거나 매우 복잡한 기술입니다.
    3. **엄격한 보안:** 접근 코드 화면(Access Screen)을 완전히 독립적인 Gate 화면으로 분리하여 우회 불가능하게 만들었습니다.


## 3. 결과물 출력 (Output Generation)

**[기존 GPTs 방식]**
- **방식:** 프롬프트 지시에 따라 1번 항목부터 13번 항목까지 하나의 거대한 채팅 응답으로 연속 출력.
- **제약:** 출력이 너무 길어 중간에 끊기는 경우가 빈번했으며("계속" 입력 필요), 문장 생성이 끝날 때까지 대기 시간이 오래 걸렸습니다. 또한 AI가 마크다운 형식을 임의로 변형하여 시각적 일관성이 자주 무너졌습니다.

**[현재 Web App 방식]**
- **방식:** 로컬 렌더링(섹션 01~04)과 AI 개별 병렬 렌더링(섹션 05~12)을 분리.
- **특장점:** 
    1. **병렬 스트리밍과 압도적 속도:** 매칭 결과(01~04섹션)는 데이터 매칭 후 즉시 표시되며, 캠페인 방향, 스토리보드, Copy 등(05~12섹션)은 여러 개의 프롬프트로 쪼개어 동시에 개별 API를 호출(`Promise.allSettled`)합니다. 단일 호출 대비 대기 시간이 혁신적으로 단축되었습니다.
    2. **출력 구조의 안정성 보장:** 응답 포맷을 강제(JSON Mode/Schema)하고 클라이언트 단에서 구조를 검증(`validateCampaignSectionData`)하므로, 양식이 절대 깨지지 않는 깔끔하고 일관된 컨테이너(카드) UI를 제공합니다.

## 4. 엔진/시스템 작동 원리 (System Logic)

**[기존 GPTs 방식]**
- **방식:** "주어진 텍스트 파일을 읽고, 입력조건과 어울리는 것을 찾아 현지어로 파생해라"라는 서술적(자연어) 지시.
- **제약:** AI가 매번 다른 결과를 도출하는 '확률론적 한계'가 뚜렷했습니다. 사용자가 제시하지 않은 스마트싱스 기기를 맘대로 추가해버리거나, Explore Contents의 핵심 구조(정체성)를 무시한 채 아예 새로운 소설을 쓰는(Hallucination) 리스크가 컸습니다.

**[현재 Web App 방식]**
- **방식:** 수학적 알고리즘(가중치 점수제) 기반의 자체 Curation Engine(`curation-engine.js`).
- **특장점:** 
    1. **정교한 가중치 매칭 교집합:** 사용자가 1인가구를 선택하면 특정 가치(예: 수면, 안심)에 +5점, 보안 장비를 선택하면 +8점 등을 부과해 점수 기반으로 가장 논리적인 시나리오를 '코드'로 낙점합니다.
    2. **AI의 '창의성' 분리 사용:** 시나리오 핵심 뼈대, 허용된 기기, 필수 Value 축은 100% 엔진 코드가 확정 짓습니다. AI는 이 확정된 재료만을 넘겨받아 '마케팅 카피라이팅', '현지 느낌의 스토리보드 묘사', 'FGD 문맥 기획' 등 상상력과 논리가 필요한 국지적인 부분만 담당합니다. 즉, '근거 없는 창작'이 불가능하도록 통제되었습니다.
    3. **결정 이유(Why) 역추적 표기:** AI의 모호성을 제거하기 위해, 어떤 신호와 타겟 가치가 겹쳐서 이 시나리오 점수(Score)가 높게 나왔는지 03. Why Selected 섹션에 명확하게 수학적 근거를 풀어 설명해 줍니다.
    4. **독립된 Nudge Microservice:** 프롬프트를 방해하지 않으면서도, 타겟 도시를 고른 직후 GPT-4o-mini가 2초안에 해당 도시의 라이프 트렌드와 마케팅 전략 기회 카드를 화면 한편에 제안해 줍니다.

---

## 5. 결론 및 종합 특장점 총정리

1. **무결점의 안정성:** AI의 가장 큰 단점이었던 환각(거짓 정보 및 권한 밖의 기기 추천)을 0%로 만들고, 출력 양식 파괴 문제 역시 0%로 종결시켰습니다.
2. **속도 극대화:** 거대한 단일 프롬프트를 8개의 작은 API 호출로 정밀하게 쪼개어 병렬 스트리밍 처리함으로써, 마케터의 체감 대기 속도를 획기적으로 낮췄습니다.
3. **편의성과 사용성 극대화:** 불편한 대화형 챗봇에서 벗어나 시각적/직관적 GUI 기반의 독립 프론트엔드를 구축했습니다. 마우스 클릭 몇 번으로 스마트싱스의 복잡한 기기 환경과 타겟 페르소나를 제어할 수 있습니다.
4. **글로벌 확장성:** 각 국가별(Locale)로 판매되는 기기 카탈로그와 언어 설정을 서버/JS 코드로 동적 관리하게 되어, 별도의 복잡한 프롬프트 수정 없이 전 세계 어느 지역에건 즉시 배포하고 대응할 수 있는 '완성형 스마트 마케팅 에이전트 소프트웨어'로 거듭났습니다.
