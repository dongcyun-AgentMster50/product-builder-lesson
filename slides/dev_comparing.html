스택의 본질이 다르다

A. 후배 방식

Make + Gemini + Drive + NotebookLM
  • 중심축: 시각적 자동화 / 오케스트레이션
  • 강점: 외부 앱 연동, 빠른 플로우 연결, 문서 기반 QA
  • 결과물: "자동화 중심 챗봇" 또는 "문서형 업무 보조 앱"
  • 주의: 사용자별 세션, 앱 권한모델, 장기 운영 구조는 우회가 늘기 쉬움

Make는 3,000+ 앱 연결과 시각적 자동화를 전면에 내세우며, 각 모듈 액션은 1 credit으로 계산됨. [1]

B. David님 방식

Firebase + GitHub + Cloudflare + AI CLI
  • 중심축: 앱 백엔드 + 배포 + 코드 기반 제품 개발
  • 강점: 상태 저장, 인증, 실시간 동기화, 릴리즈/롤백
  • 결과물: "서비스형 웹앱" 또는 "제품형 SaaS"
  • 주의: 초기 설계·구현량이 더 큼

Firestore는 실시간 동기화, 오프라인, 보안 규칙 기반 데이터 계층을 제공하고, Cloudflare Pages는 GitHub PR마다 preview URL을 생성함. [2][3]

A는 "자동화 플랫폼을 앱처럼 쓰는 접근"이고, B는 "앱 플랫폼으로 실제 웹앱을 만드는 접근"이다.

발표 포인트: 둘 다 채팅형처럼 보이지만, 실제로는 서로 다른 계층의 조합1 / 9

개발 단계별 비교

비교 단계A. Make 기반B. Firebase 기반
기획/PoC매우 빠름. 문서 연결 + 자동화 흐름 시연이 쉬움. 데모 적합.초기 구조 설계 필요. 대신 다음 단계로 이어지는 재사용성이 높음.
구현화면보다 플로우 구현에 강함. UI 자유도는 낮음.프론트/백/API를 분리 설계 가능. UX 자유도 높음.
테스트시나리오 단위 점검은 빠르지만 회귀 테스트와 버전 흐름이 약함.GitHub 브랜치, PR, preview URL, 코드 리뷰 중심 운영이 쉬움. [3]
배포자동화 시나리오 publish는 간단. 하지만 앱 자체 배포는 별도 고민 필요.Cloudflare Pages/Workers로 정석적인 웹앱 배포가 가능.
운영/관측운영 흐름이 SaaS 정책과 credit 구조에 강하게 의존.앱 로그, DB, 권한, 배포 체계를 분리 운영 가능.
리팩터링초반엔 빠르지만 플로우가 복잡해질수록 구조가 꼬일 위험.초반 비용은 크지만 장기 구조 수정과 확장에 유리.

비용 비교: 런타임과 개발도구를 분리해서

A. 런타임 기본비

낮은 진입 · 높은 사용량 민감도
  • Make Free: 1,000 credits / month, Core는 $9 / month (10k credits). [1]
  • Google AI Pro: $19.99 / month, NotebookLM higher access 포함. [4]
  • Gemini API: 2.5 Flash-Lite 입력 $0.10 / 1M tokens, 출력 $0.40 / 1M tokens. [5]
  • 의미: 초반은 싸게 시작 가능하지만, 채팅 1턴이 여러 Make 모듈을 거치면 credit 누적이 빨라짐.

B. 런타임 기본비

초기 런타임 저비용 · 구조 확장 유리
  • Cloudflare Workers Paid: 최소 $5 / month, 요청·로그 등 사용량 기반. [6]
  • Firestore Free quota: 1 GiB 저장, 50k reads/day, 20k writes/day. [7]
  • Firebase Auth Spark: instrumentless usage 3,000 DAU/day. [8]
  • 의미: 저트래픽 제품형 웹앱은 런타임이 매우 싸게 시작될 수 있음.

셋업 비용 vs 유지관리 비용

A. 후배 방식

셋업 낮고, 구조 복잡화 시 유지비 상승
  • 셋업: 낮음 — no-code / low-code 연결이 빠름
  • 초기 개발기간: 짧음
  • 유지관리: 중~높음 — 시나리오가 늘수록 흐름 파악과 수정 영향도 관리가 어려워질 수 있음
  • 장기 리스크: 앱 권한/세션/버전/테스트 요구가 커질수록 재설계 가능성 증가

초기 진입장벽은 낮지만, "앱 본체" 요구가 커질수록 유지관리 난이도가 상승하는 유형

B. David님 방식

셋업 높지만, 유지비 안정적
  • 셋업: 중간 이상 — 데이터 구조, 인증, 배포 구조 설계 필요
  • 초기 개발기간: 더 김
  • 유지관리: 낮음~중간 — 코드/브랜치/PR/preview 흐름으로 관리 가능
  • 장기 강점: 기능 추가, A/B 테스트, API 확장, 역할 분리에 유리

처음엔 무겁지만, 제품이 커질수록 구조적 이점이 쌓이는 유형

핵심: "초기 셋업 비용"만 보면 A, "총소유비용(TCO)"까지 보면 B가 유리5 / 9

최종 추천: 어떤 방식이 더 좋은가?

정답 1

"빨리 검증"이 목표면 A

사내 문서 기반 답변, 간단한 챗 UI, 외부 앱 자동화를 짧은 시간에 연결해야 한다면 A가 더 빠르다.

정답 2

"제대로 운영할 앱"이면 B

브랜드 웹앱, 사용자 상태, 권한, 릴리즈, 장기 기능 확장이 필요하면 B가 더 낫다.

베스트 프랙티스

권장 아키텍처 = B를 본체로, A를 보조 자동화 레이어로
  • 웹앱 본체: Firebase + GitHub + Cloudflare
  • 세션/대화 저장: Firestore
  • 배포/리뷰: GitHub + Cloudflare preview
  • 외부 SaaS 자동화: Make
  • 문서 리서치/보조 QA: Drive + NotebookLM
  • 즉, 주요 채팅턴은 B, 저빈도 자동화 액션은 A

최종 판단: David님 방식이 장기적으로 더 좋은 선택. 단, 초기 검증 단계에서는 A의 속도를 전략적으로 활용 가능.
