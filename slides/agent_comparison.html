1. The Context
2 / 15
도입 배경

왜 우리는 기존 GPTs 방식에서 벗어나야 했나?

프롬프트 엔지니어링의 한계를 뼈저리게 체감한 후, 구조 자체를 바꿔야 생존할 수 있었습니다.

무엇을 위한 부분인가?

Purpose of Change
마케터들이 챗봇과 "싸우는" 시간을 줄이기 위함입니다. GPTs 환경에서는 긴 지시사항이 누락되거나, 환각 증세(엉뚱한 기기 추천 등)가 빈번하게 발생하는 "플랫폼 구조상의 한계"가 명확했습니다. 이를 극복하려면 코드가 통제하는 웹 앱 환경이 절실했습니다.

결과 및 활용 방향

Outcome
AI 모델 자체를 통제할 수는 없지만, AI가 개입하는 "시점"과 "경로"를 통제할 수는 있습니다. 마케터는 이제 AI가 지시를 잊어버릴까 걱정할 필요 없이 정해진 인터페이스를 자유롭게 활용하면 됩니다.

2. Architecture: GPTs

아키텍처 제약사항

기존 GPTs 방식의 구조적 한계 (RAG 및 프롬프트 구동)

자연어(Natural Language)만으로 모든 논리를 제어하는 것은 매우 불안정합니다.

무엇을 위한 부분인가?

Current Limitations
기존 챗봇 플랫폼은 사용자가 문서를 올리면 이를 RAG(문서 파편 검색)로 찾아서 조합합니다. 로직 파트(무엇을 매칭할지)와 출력 파트(어떻게 글을 쓸지)가 하나의 AI 뇌통 속에서 섞여버려, 엄격한 "필터링" 이나 "가중치 점수 메기기"를 기대하기 힘들었습니다.

그 결과 생성되는 문제점

Unreliable Outcomes
동일한 입력을 넣어도 기분이 내키는 대로 다른 시나리오를 꺼내오거나, 지원하지 않는 신규 AI 기능을 마치 있는 것처럼 꾸며내는 '확률론의 저주'를 겪게 됩니다. 마케터는 출력물을 매번 100% 팩트체크해야만 했습니다.

3. Architecture: Web App

아키텍처 진화

독립 Web App 구축: 매칭과 생성의 엄격한 분리

AI는 글쟁이일 뿐, 판사가 되어서는 안 됩니다. 매칭 권한을 AI로부터 박탈했습니다.

어떻게 개선했는가?

Hardcoded Logic
Node.js와 브라우저 단에서 작동하는 "알고리즘"이 매칭 판사 역할을 전담합니다. 사용자의 입력을 받으면 가중치 점수(+5점, +8점 등)를 엑셀 수식처럼 정확하게 매겨서 1위부터 5위까지의 시나리오를 고정 확정(Lock-in) 짓습니다.

가장 중요한 핵심 결과

Zero Hallucination core
AI(LLM)는 오직 "확정된 뼈대"만을 전달받아, 그 내용에 살을 입히고 현지의 느낌을 살려 카피를 번역하거나 묘사하는 "표현 업무"에만 집중하게 됩니다. 즉, 팩트 오류 확률이 기술적으로 거세되었습니다.

Enterprise Necessity

Friction Point
기존엔 사용자가 직접 대화창에 "1번 마케터, Q2 미국, Q3 대학생, Q4 TV"라고 키보드로 입력해야 진행이 되었습니다. 또한 대화창 특성상 대충 치거나 오타를 내면 AI가 자기 맘대로 해석하고 단계를 건너뛰어 버립니다.

결과 및 마케터의 불편함

User Fatigue
진행 흐름이 툭툭 끊기고 원하지 않는 대답이 나오면 처음부터 프롬프트를 다시 치며 어르고 달래야 했습니다. 바쁜 일선 마케터들에겐 지속적으로 사용할 유인을 떨어뜨리는 강력한 허들(Friction)이었습니다.

어떻게 사용되는가?

GUI Magic
화면에 보이는 페르소나 카드들(예: 애완동물, 수면 부족)을 클릭으로 껐다 켤 수 있으며, 타겟 기기 역시 이미지 배열 속에서 '장바구니' 담듯이 체크하면 그만입니다. 또한 필수값을 누르지 않으면 다음 버튼 자체가 비활성화됩니다.

결과: 휴먼 에러의 근절

Strict Data Structure
클라이언트(브라우저)에서 완벽히 정제된 JSON 데이터 포맷만이 서버로 넘어오므로, 뒤에 이어질 매칭 알고리즘이 언제나 100% 동일하고 정확하게 구동될 수 있는 완벽한 기초 데이터를 얻습니다.

10. Output: Serial vs Parallel

출력 생성 차이 - 1

직렬(Serial) 거북이 출력 vs 병렬(Parallel) 초고속 스트리밍

마케터의 가장 큰 피로감은 모니터를 멍하니 바라보는 대기 시간이었습니다.

기존 GPTs 방식

Sequential monolithic
AI에게 "1. 요약, 2. 기기선정, 3. 카피라이팅 ... 13. 기대효과까지 다 써줘"라고 명령하면, AI는 1번을 쓰기 시작해 13번을 끝낼 때까지 계속 한 줄씩 느리게 타자를 칩니다. 도중에 토큰이 모자라면 끊어져서 '계속'을 눌러야 했습니다.

신규 Web App 방식

Parallel `Promise.all`
앱의 서버는 13개 항목을 조각낸 뒤, "1번만 알려줘", "5번만 알려줘", "11번만 알려줘"라는 요청을 API 8개로 분할해 동시에 동시에 서버로 쏘아 올립니다. 화면 곳곳의 박스가 일제히 타자를 치며 순식간에 빈 공간을 채워 넣습니다.

Conclusion

단순 챗봇에서 진정한 '엔터프라이즈 솔루션'으로

지금껏 살펴보았듯 어플리케이션(Application)이 지니는 4대 강점은 명확합니다.

1. 철벽 보안 & 제로 환각

Absolute Integrity
단도직입적인 접근 코드 관리를 통해 회사 자산 통제. Curation 수학 엔진을 사용해 AI의 멋대로 파생 예측(Hallucination) 징후들을 철저히 코드 차원에서 절제가능.

Time to Value
