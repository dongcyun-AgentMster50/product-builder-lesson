Architecture Overview

전반적인 아키텍처 및 핵심 솔루션

순수 웹 기술 기반의 독립형 환경(웹 앱)으로 구축되어, 속도와 보안, 정확성을 일거에 달성했습니다.

시스템 구성

Tech Stack
  • Frontend: 바닐라 JavaScript(ES6+), HTML5, 순수 CSS로 구축된 SPA(Single Page Application). 체감 속도가 매우 뛰어남.
  • Backend: Node.js `http` 모듈 기반 초경량 서버. 정적 배포 및 API Gateway (CORS 회피/Key 은닉) 역할 동시 수행.
  • Flow: 4단계 직관적 마법사(Wizard) UI 인터페이스 기반 작동.

연동 솔루션 및 API

AI & Gateway
  • Multi-LLM: 상황에 따라 OpenAI `gpt-4o`, `gpt-4o-mini`, Anthropic `claude-sonnet` 시리즈를 혼합 사용. (Nudge는 빠른 속도 중시, Output은 추론 품질 중시).
  • SSE 통신: AI 응답은 Server-Sent Events(SSE) 기술로 브라우저에 타자를 치듯 실시간 스트리밍되어 체감 대기 속도를 소거.

  • 결과: 판매하지 않는 제품이 시나리오에 포함되는 오류를 기술적으로 원천 방지함.

3. Curation Engine
4 / 10
순수 알고리즘 매칭

핵심 모듈: 큐레이션 엔진 (curation-engine.js)

AI의 확률적 추론에 기대지 않고, 100% 자바스크립트 수식을 사용해 DB(`explore_db.json`)와 정확히 매칭합니다.

처리 단계함수 / 메커니즘
1. 태그 변환 `buildExploreTagsFromInput()`
페르소나와 기기, 목적 텍스트를 파싱하여 ["가족 케어", "수면"] 같은 가중치 태그 배열로 변환합니다.
2. 점수 산정 `scoreScenario()`
입력 태그와 기존 DB 시나리오가 겹치면 가중치 점수를 수학적으로 부여합니다. 특히 겹치는 기기(Devices)가 많을수록 보너스 점수가 가중됩니다.
3. 최적 선별 `curateScenarios()`
최종 점수별로 랭킹을 매긴 후, 중복을 제거한 Top 시나리오를 선별해 결과 객체 배율로 리턴합니다.
AI가 임의로 스토리를 창작하며 발생하던 환각(Hallucination) 리스크를 제로(0%)로 만듭니다. 시나리오는 코드가 뽑고, AI는 포장만 합니다.

캠페인 생성 파이프라인

병렬 스트리밍 아키텍처 (campaign-output.js)

13개 영역의 결과물을 하나의 기다란 AI 응답으로 받지 않고, 목적에 따라 로컬과 분할 병렬 랜더링으로 나누었습니다.

순수 알고리즘 렌더링
(Sections 01~04)

0초 생성 영역
  • AI Delay Zero: Curation Engine의 연산 결과를 즉각 UI로 렌더링합니다.
  • 논리적 역추적: 해당 시나리오가 "왜 추천되었는지", "어떤 기기가 겹치는지" 등을 역산출하여 _buildWhyExplanation() 함수로 표시.

병렬 멀티 스트리밍
(Sections 05~12)

Promise.allSettled
  • 개별 프롬프트 발송: 05부터 12섹션까지 각기 다른 전문 프롬프트(카피/스토리/FGD 등)를 동시에 백엔드 API로 전송합니다.
  • UI 부분 렌더링: reader.read()를 활용한 SSE 통신으로, 응답이 돌아오는 섹션 카드부터 화면에 글씨가 실시간 채워집니다.
API Gateway: server.js를 통해 OpenAI/Anthropic 엔드포인트 연동 및 JSON Schema 강제 출력 조율

Architecture vs GPTs

구성 요소 및 아키텍처 비교

프롬프트 하나에 모든 동작을 맡기던 과거 방식에서, 코드가 코어 논리를 통제하는 방식으로 진화했습니다.

기반 기술
기존 GPTs 방식ChatGPT 플랫폼 내 챗봇 (자연어 프롬프트). 지시사항의 길이가 길어져 Context Window 한계 노출.
현재 Web App 방식 (승)Node.js Backend + Vanilla JS Frontend. 완전한 독립 애플리케이션으로 커스텀이 자유로움.
설계 구조
기존 GPTs 방식Part 1~7로 나뉜 방대한 텍스트 인스트럭션 기반. DB는 RAG 방식으로 검색 의존.
현재 Web App 방식 (승)텍스트 제약 탈피. Curation Engine이 100% 자바스크립트 수식으로 DB를 직접 파싱하고 통제.
환각 위험
기존 GPTs 방식RAG 시스템 특성상 정해진 DB를 무시하거나 임의로 결론지어버리는 문제 빈번.
현재 Web App 방식 (승)매칭 영역과 텍스트 생성 영역을 완전히 분리하여 사실 오류 원천 차단 (Zero Risk).

User UX vs GPTs

마케터 체감 경험 및 UI 비교

대화형 인터페이스(Chat UI)의 모호함을 시각적 GUI 마법사로 완전히 대체했습니다.

기존 GPTs 챗봇 방식

Typing intensive
  • 타이핑 피로도: "Q1은 리테일, Q2는 미국..." 식으로 수동 타이핑 진행.
  • 진행 통제 불가: AI가 중간에 멋대로 단계를 건너뛰어 곧바로 결과물을 내는 오류 일상다반사.
  • 오타/예외 취약: 잘못 입력했을 때 정규화 과정 없이 엉뚱한 결과물 연동 우려.

현재 Web App 방식

Click intensive
  • 시각적 마법사: 국가 드롭다운, 태그 체크리스트 등 시각적 버튼/카드를 통한 완전한 GUI 플로우 설계.
  • 동적 대응력: 국가 선택 순간 사용 가능한 카탈로그(Products)가 실시간 필터링되어 노출됨. (챗봇으로는 구현 불가능).
  • 강력한 인증: Access Screen을 외부 단절 게이트로 분리해 보안 통제권 완벽 장악.

Logic Engine vs GPTs

엔진 시스템 작동 원리 비교

AI의 '창의성(Creativity)'을 모호성 통제 불가(Hallucination)가 아닌, 필요한 영역에만 배치하여 안정성을 확보했습니다.

기존 시스템 (확률론적 엔진)

Probabilistic Guessing
  • RAG 의존: 주어진 파일을 읽고 알아서 조합하라는 서술적 프롬프트 사용.
  • 매번 다른 논리 산출: 질문할 때마다 AI의 판단이 달라져 객관적인 퀄리티 측정이 제한됨. 사용자가 묻지도 않은 이상한 기기를 멋대로 끼워 팜.

웹앱 시스템 (결정론적 알고리즘)

Deterministic Engine
  • 가중치 알고리즘 매칭: 1인가구 선택(+5), 보안가치 선택(+8) 등 정확한 포인트 합산 로직을 타고 코드가 정답을 落點.
  • AI의 분리 사용 (통제된 창의성): 로직이 정해준 핵심 뼈대 안에서만 AI가 "카피라이팅", "로컬 뉘앙스 번역" 등만 진행하게 제어함.
  • 추가 기능으로 타겟 도시를 고른 직후 GPT-4o-mini가 2초안에 Nudge Microservice 팝업 전략 지원.
