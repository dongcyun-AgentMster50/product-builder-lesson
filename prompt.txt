**Purpose**: Show the user experience concretely in a structured storyboard-like format. Use Explore 2.0 logic, but organize this section as mini-scenarios built from ① Customer Pain Point ② Function / Resolution ③ Customer Benefit.

**Required blocks**:

**2-1. Target Customer**
- One line only
- Format: `[Country/City] / [Segment] / [Lifestyle context]`

**2-2. Applied SmartThings Service Scenario**
- One line only
- Format: `[Service A] / [Service B] / [Service C] service scenario`
- Use 2–3 services max

**2-3. Scenario Cases (2–3 cases)**
- Build 2 or 3 mini-scenarios
- Each case title must reflect one of the actual values selected or inferred for this scenario
- Use 1–2 values only when that is what the scenario truly reflects; do not force all four values
- Recommended title format:
  - `[Care] ...`
  - `[Secure] ...`
  - `[Save] ...`
  - `[Play] ...`

For each case, output exactly these 3 blocks in order:

**① Pain Point**
- A concrete daily-life discomfort, anxiety, or unmet need

**② 기능/해결책**
- Explain how SmartThings / AI Home / connected devices solve that situation
- Use practical feature language, not abstract product marketing
- Conditional wording if support is unverified

**③ 고객 Benefit**
- What improves for the customer in plain language
- Focus on the felt result, not feature repetition

**Writing rules**
- Make each case easy to scan and presentation-ready
- No JSON, no JTBD sentence, no table in this section
- Keep the scenario grounded in lived behavior and visible benefit
- Each case should feel like one compact story card that can be shown in a slide
- This section replaces the old time-sequence-heavy detailed scenario. Prioritize situation-based value proposition over chronological journey mapping.
- Cases should still imply how the user moves toward real use, not remain at an idea-only level
