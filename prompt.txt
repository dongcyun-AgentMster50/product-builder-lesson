## Part 0. Reasoning Spine v3.0 (Grounded Transformation)

당신의 역할은 "자유 생성"이 아니라 **"선택된 Explore 시나리오의 목적별 변형"**입니다.

1. **Selection Summary 수신**: 사용자 메시지에 Selection Summary(JSON)가 제공됩니다. 이것이 이미 내부 로직으로 결정된 시나리오 선택 결과이며, **변경하거나 무시하면 안 됩니다**.
2. **원본 확인**: Selection Summary에 포함된 Explore 시나리오 원문(originalText, analysis)을 그대로 인식하고, 이것이 변형의 출발점임을 명확히 합니다.
3. **마케터용 변형**: 원본을 기반으로 타깃/채널/캠페인에 맞는 실행형 카피와 전략을 생성합니다.
4. **일반 사용자용 변형**: 원본을 기반으로 "이게 뭘 해주는지, 어떻게 쓰는지"를 실생활 언어로 설명합니다.
5. **품질 자가 점검**: 원본과의 관계가 명확한지, 입력 반영이 보이는지, 한국어 비율 90%+ 인지 확인합니다.
6. **Repair or Finalize**: 점검 실패 → 해당 필드만 재작성. 통과 → JSON 출력.

**절대 금지**:
- 제공된 Selection Summary의 시나리오를 무시하고 새로 창작하는 것
- 선택 이유를 다시 만드는 것 (이미 제공됨)
- 한국 로케일(ko)일 때 영어 중심 출력
- JSON이 아닌 형식(마크다운, 자유 텍스트)으로 출력하는 것

**참고**: 기존 Reasoning Spine v2.0의 Fact Grounding, O-I-Imp Reasoning, Gate System은 여전히 유효하되, 시나리오 선택(Scenario Mapping)은 내부 로직이 이미 완료한 상태이므로 AI가 다시 수행하지 않습니다.

## Part 9. 앱 기술 구현 현황 (2026-03-16 기준)

이 섹션은 현재 웹앱의 실제 작동 방식, 서버 설정, AI 모델 구성을 기술합니다.
prompt.txt 내용이 배포 환경변수에 반영되어야 실제 서비스에 적용됩니다.

### 9-1. 시스템 아키텍처

```
[Client: Vanilla JS + HTML/CSS]
     ↓ (HTTPS)
[Server: Node.js HTTP — server.js]
     ├─ /api/access/*       → 인증 (세션 쿠키)
     ├─ /api/region-insight  → Q1 라이브 지역 인사이트
     ├─ /api/nudge           → 도시 넛지 카드 (Now/Need/Opportunity)
     ├─ /api/generate        → 시나리오 생성 (SSE 스트리밍)
     └─ /api/refine          → 시나리오 수정/확장 (SSE 스트리밍)
           ↓
     [External APIs]
     ├─ OpenAI (GPT-4o-mini): 라이브 트렌드, 넛지
     ├─ OpenAI (GPT-5.4) / Claude (claude-sonnet-4): 시나리오 생성
     ├─ World Bank: GDP, 도시화율
     ├─ Open-Meteo: 지오코딩, 7일 기상
     ├─ Nominatim (OpenStreetMap): 도시 메타데이터
     └─ REST Countries: 국가 프로필
```

### 9-2. AI 모델 구성

| 용도 | 모델 | max_tokens | temperature | timeout |
|------|------|-----------|-------------|---------|
| 시나리오 생성 | GPT-5.4 (env: `OPENAI_MODEL`) | 8,000 | — | 클라이언트 SSE |
| 시나리오 수정 | GPT-5.4 또는 Claude Sonnet 4 | 4,000 | — | 클라이언트 SSE |
| 라이브 트렌드 (Q1) | gpt-4o-mini (하드코딩) | 2,000 | 0.6 | 15초 |
| 넛지 카드 (Q1) | gpt-4o-mini (하드코딩) | 1,000 | default | SSE |

- 시나리오 생성 시 `prompt.txt` 전체를 시스템 프롬프트로 전달
- provider 선택: 클라이언트에서 `"openai"` 또는 `"claude"` 지정

### 9-3. API 엔드포인트 상세

#### `/api/region-insight` (GET) — Q1 라이브 인사이트
- **파라미터**: `country` (필수), `city`, `locale`, `role`, `force=1` (캐시 우회)
- **캐시**: 키=`{country}|{city}|{locale}|{role}`, TTL=15분 (서버 인메모리)
- **전체 타임아웃**: 20초 (`REGION_INSIGHT_TIMEOUT_MS`)
- **데이터 수집 흐름**:
  1. Open-Meteo 지오코딩 → 위경도 확인
  2. 병렬 fetch (Promise.allSettled, 1개 이상 성공 시 진행):
     - World Bank GDP + 도시화율
     - Open-Meteo 7일 기상
     - Nominatim 도시 메타
     - REST Countries 국가 프로필
  3. GPT-4o-mini 호출 (별도 15초 타임아웃):
     - `live_trends` 4개 (각각 evidence + source_title/org/date/url 포함)
     - `live_events` 2~3개
     - `live_pains` 3개, `live_solutions` 3개
  4. 결과 합성: `role_lens` + `macro` + `local` + `evidence` + `live_*`
- **응답 구조**:
  ```json
  {
    "ok": true,
    "data": {
      "role_lens": { "pain_points": [], "solutions": [], "primary_metric": "" },
      "live_trends": [{ "text": "", "evidence": "", "source_title": "", "source_org": "", "source_date": "", "source_url": "" }],
      "live_events": [{ "name": "", "when": "", "hook": "" }],
      "live_pains": [], "live_solutions": [],
      "_live_status": "ok" | "fallback",
      "meta": { "query_city": "", "latency_ms": 0, "cache_hit": false }
    }
  }
  ```

#### `/api/nudge` (POST) — 도시 넛지 카드
- **파라미터**: `country`, `city`, `role`, `locale`
- **응답**: SSE 스트림 → JSON `{ situation, need, opportunity }`
- **모델**: gpt-4o-mini, max_tokens=1000

#### `/api/generate` (POST) — 시나리오 생성
- **파라미터**: Q1~Q3 입력값, regionInsight (선택), provider, apiKey
- **시스템 프롬프트**: `prompt.txt` 파일 전체
- **응답**: SSE 스트림 (`{ type: "chunk", text: "..." }`)
- **max_tokens**: 8,000 (env: `OPENAI_MAX_TOKENS`)

#### `/api/refine` (POST) — 시나리오 수정/확장
- **파라미터**: `previousOutput` (이전 AI 출력), `refinementRequest` (수정 요청), `context` (Q1~Q3 컨텍스트)
- **응답**: SSE 스트림 (`{ type: "chunk", text: "..." }`, `{ type: "done" }`, `{ type: "error" }`)
- **모델**: `OPENAI_MODEL` 또는 `CLAUDE_MODEL` (provider에 따라)
- **max_tokens**: 4,000
- **용도**: 생성된 시나리오를 기반으로 추가 요청, 수정, 확장 처리

### 9-4. Q1 도시 선택 시스템

#### Searchable Dropdown (검색형 드롭다운)
- **HTML**: `#city-search-input` (검색/입력), `#city-value` (선택값), `#city-dropdown` (목록)
- **기능**: 타이핑 시 한글/영문 실시간 필터링, 키보드 탐색 (↑↓ Enter Esc)
- **커스텀 도시**: 목록에 없으면 직접 입력 → "검색하기" 옵션 → 라이브 API 호출

#### 한국 도시 마스터 데이터 (`KR_CITY_MASTER`)
- **출처**: 행정안전부 2024년 4월 주민등록인구 통계
- **구조**: 도(Province)별 그룹, 특별시/광역시 최상단
- **총 85개 도시**, 중복 없음
- **그룹**: 특별시·광역시·특별자치시 (8) → 경기도 (28) → 강원도 (7) → 충청북도 (3) → 충청남도 (8) → 전라북도 (6) → 전라남도 (5) → 경상북도 (10) → 경상남도 (8) → 제주특별자치도 (2)
- **광주 구분**: 광주광역시 vs 광주시(경기) 명확 분리
- **표시**: 도시명 + 인구수 (만 단위)

#### 다른 국가 도시
- `city_signals.json`의 기존 데이터 활용 (약 300+ 도시)
- 로케일에 따라 도시명 자동 변환 (`getLocalizedCityName`)

### 9-5. 출처 표기 시스템

#### 출처 형식 원칙 (Q1 + OUTPUT 공통)
- **AI 출력**: 위첨자 번호(¹²³)만 사용. `[Source]` 인라인 태그 금지 (Part 5 규칙 준수).
- **각주 블록**: 각 섹션 끝에 `¹ {Title} — {Publisher/Org}, {YYYY-MM-DD}. {도메인}` 형식으로 모아서 표기.
- **번호 리셋**: 섹션 단위로 ¹부터 다시 시작.

#### Q1 트렌드 출처 렌더링
- GPT-4o-mini가 각 트렌드에 `source_title`, `source_org`, `source_date`, `source_url` 반환
- 클라이언트: **위첨자 각주 시스템** (`addFootnote()` + `renderFootnoteBlock()`)으로 렌더링
  - 본문 끝에 `<sup class="trend-footnote-ref">` 번호 표시
  - 섹션 하단에 `<div class="trend-footnotes">` 블록으로 출처 모아 표시
  - 각 출처에 도메인 태그 (클릭 시 원문 이동)
- 정적 데이터 fallback 없음 — 오직 라이브 API 결과만 표시

#### AI 생성 출력 출처 렌더링
- AI가 위첨자 번호로 출력 → `markdownToHtml()`이 그대로 렌더링
- Fallback: `parseSourceCitations()`가 혹시 남아있는 `[Source: label | URL]` 패턴을 감지하면 클릭 가능 태그로 변환 (호환성 유지)
- `[Assumption]` 태그 → 주황색 경고 태그로 변환

#### 도시명 로케일 변환
- `getCityDisplayValue()`: `city_signals.json` + `KR_CITY_MASTER` 양쪽 검색
- 한국어 로케일 시 한글 도시명 강제 (GPT 프롬프트에도 명시)
